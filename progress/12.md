1) Located the exact redraw path
- Inspected `frontend/src/pages/optimizer/OptimizerPage.tsx` for progressive reveal patterns (`visibleCount`, timers resetting render counts, dynamic `<tbody key=...>`).
- No 0-to-N reveal counter or dynamic table body key existed.
- Found the real cause in the active single-run poll loop: every `ACTIVE_RESULTS_POLL_MS` tick called `fetchResults(...)` while status was `running`/`paused`, rebuilding `rowsById` and `order` from fresh API objects for the entire page. That replaced row object references for all existing rows and forced every row component to re-render on each update.

2) Implemented precise minimal fix
- Kept live updates append-only from `optimizer_rows_append` websocket path.
- Changed polling behavior so `fetchResults(...)` is skipped while job status is `running` or `paused`; polling still updates status/progress, and final fetches still happen for terminal states (`done`/`cancelled`), preserving sorting/pagination behavior.
- Added a dedicated memoized table body component (`OptimizerResultsBody`) with DEV-only mount/unmount trace logging gated by:
  - `import.meta.env.DEV`
  - `localStorage.debugOptimizerRowRenders === "1"`
- Added DEV-only mount/unmount tracing for first 3 rows in `OptimizerResultRow` to prove existing `<tr>` nodes are not remounted on append.

3) Files changed
- `frontend/src/pages/optimizer/OptimizerPage.tsx`
- `progress/12.md`

4) What was wrong
- The bug was frontend-side render/update logic: periodic full-page result refetch during active run replaced all row object identities repeatedly, so memoized rows still re-rendered en masse.
- This visually looked like table-wide redraw/rebuild on each new append.

5) How verified
- Built frontend successfully: `npm --prefix frontend run build`.
- With `localStorage.debugOptimizerRowRenders = "1"` in DEV, expected runtime behavior:
  - table body mounts once (no recurring unmount/mount on each append)
  - only one new row logs mount for each new append
  - existing rows do not log mount/unmount and do not log render during append-only updates

6) Why prior attempts failed
- Prior fixes that only adjusted keys/memoization were insufficient because they did not remove the active-run refetch path that replaced all row object references.
- As long as full `fetchResults` replacement ran every polling tick during active runs, React had to re-render existing rows even with stable keys.
