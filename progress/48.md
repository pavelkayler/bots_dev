1) What changed per file
- `backend/src/runtime/configStore.ts`: removed the `paper.entryTimeoutSec` schema upper bound by changing validation from `.min(1).max(3600)` to `.min(1)`.
- `backend/src/api/http.ts`: added `/api/config` POST normalization for `paper.entryTimeoutSec` so incoming values are sanitized to a finite integer with lower bound 1 (`Math.max(1, Math.floor(raw))`) and no upper clamp.
- `frontend/src/pages/optimizer/OptimizerPage.tsx`: updated optimizer row “Copy” handler to map `row.params.timeoutSec` into paper patch `entryTimeoutSec` only when finite (seconds -> seconds), otherwise omit it so existing draft value remains unchanged.
- `frontend/src/features/config/components/ConfigPanel.tsx`: updated pending optimizer patch application to actually merge `patch.paper.entryTimeoutSec` into `draft.paper.entryTimeoutSec`, ensuring copied timeout appears in Config > Paper.

2) Why
- Optimizer `timeoutSec` needs to apply 1:1 to paper `entryTimeoutSec`.
- A max cap on `entryTimeoutSec` prevented large optimizer values from being applied.
- Removing backend cap and avoiding frontend/backend clamping enables large timeout propagation while preserving sane numeric normalization.

3) How to verify locally
- Build both projects:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- In Config UI, set Paper `entryTimeoutSec` to `100000`, click Apply, verify request succeeds (no 400).
- In Optimizer results, choose a row with large `timeoutSec`, click Copy, open Config and confirm Paper `entryTimeoutSec` equals copied value, then Apply succeeds.
- Restart app and call `GET /api/config`; confirm `config.paper.entryTimeoutSec` persisted as expected.

4) Notes
- Very large `entryTimeoutSec` means entry orders may remain pending for a long time before expiring. This is intended by current requirements.
