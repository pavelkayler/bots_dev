1) Where entryTimeoutSec was limited
- The optimizer "Copy" mapping in `frontend/src/pages/optimizer/OptimizerPage.tsx` was not a strict 1:1 transfer because it wrote `entryTimeoutSec` through `quantizeByPrecision(...)` and only read from `row.params.timeoutSec`.
- Backend and config UI were reviewed for upper caps: `backend/src/runtime/configStore.ts` validates `paper.entryTimeoutSec` with `.min(1)` and no `.max(...)`, and `frontend/src/features/config/components/ConfigPanel.tsx` `entryTimeoutSec` input has no upper `max` attribute/clamp.

2) What changed
- Updated optimizer "Copy" handler to map timeout seconds directly (`paper.entryTimeoutSec = row.params.timeoutSec`), with fallback support for `row.timeoutSec` when needed.
- Removed precision quantization from this mapping so timeout is copied seconds-to-seconds exactly.
- Kept `rearmDelayMs`/`rearmSec` no-limit mapping behavior unchanged.

3) How to verify
- Build backend: `npm --prefix backend run build`.
- Build frontend: `npm --prefix frontend run build`.
- In Config → Paper, set `entryTimeoutSec=100000` and click Apply; request should succeed (no 400/409).
- In Optimizer results, click Copy on a row with large `timeoutSec`; Config → Paper `entryTimeoutSec` should match that value exactly.
- Apply, then refresh/restart and check `/api/config` to confirm persisted `paper.entryTimeoutSec`.

4) Notes
- Very large `entryTimeoutSec` means pending entry orders can remain live for a long time before expiry; this is intended.
