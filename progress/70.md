# 70 - Stage 11 apply-next-trades + 5m/10m optimizer window support

Date: 2026-03-06

## 1) Runtime apply-for-next-trades

### Problem
`POST /api/config` always reported paper config as `next_session`, so operationally there was no explicit in-run apply path for future entries.

### Changes
Files:
- `backend/src/runtime/runtime.ts`
- `backend/src/paper/PaperBroker.ts`
- `backend/src/demo/DemoBroker.ts`
- `backend/src/api/http.ts`

Implemented:
1. Runtime method `applyConfigForNextTrades(...)`.
2. Broker methods:
   - `PaperBroker.applyConfigForNextTrades(...)`
   - `DemoBroker.applyConfigForNextTrades(...)`
3. `POST /api/config` now:
   - when runtime is active, applies trading params to brokers for subsequent entries
   - returns `applied.paper = "next_trades"`
   - otherwise keeps `applied.paper = "next_session"`.

Safety note:
- Existing open position state is not rebuilt; updates are consumed by new entry attempts.

## 2) Optimizer 5m/10m + minute-path behavior request

### Problem
Optimizer was effectively constrained to 15m+.

### Changes
Files:
- `backend/src/optimizer/runner.ts`
- `backend/src/api/http.ts`
- `frontend/src/features/optimizer/utils/inputValidation.ts`
- `frontend/src/features/optimizer/utils/timeframes.ts`
- `frontend/src/pages/optimizer/OptimizerPage.tsx`

Implemented:
1. Lowered optimizer minimum timeframe from 15 to 5 minutes.
2. Enabled 5m and 10m options in UI.
3. Updated frontend validation message/rule (`>= 5`).
4. Updated backend run validation (`optTfMin >= 5`).
5. Replay behavior kept compatibility:
   - uses minute cache when present
   - falls back to dataset interval cache when 1m is unavailable.

This preserves the intended behavior:
- if minute points exist, strategy windows (5m/10m) use underlying minute path
- if minute layer is unavailable, replay uses legacy coarser interval source.
