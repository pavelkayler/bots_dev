# Progress Report 15

## Root cause
`minTrades` was being applied during optimizer row ingestion. In loop mode, incoming WS/poll/completed rows were filtered before insertion, so batches where `trades=0` were discarded entirely. This kept the in-memory store empty even while WS batches were arriving, causing the UI to show "No result rows received yet" incorrectly.

## Changes made
- Updated `frontend/src/pages/optimizer/OptimizerPage.tsx`:
  - Removed `minTrades` filtering from loop ingestion in `upsertLoopRunRowsAppend`; all incoming rows are now stored and deduped by signature.
  - Kept append-only behavior for loop stores (no clearing, only append unseen row IDs).
  - Updated loop WS debug log (`[optimizer-rows-append-loop]`) to report:
    - `rawBatchSize`
    - `minTradesLimit`
    - `displayEligibleCount`
    - `appendedCount` computed from raw rows
    - `totalBefore` / `totalAfter`
  - Applied `minTrades` as a display-only filter when deriving table rows (`rowsForDisplay`), so underlying stored rows remain untouched.
  - Updated no-rows warning logic to use raw stored row count for current context (loop or single) rather than displayed row count.
  - Added a small hint when raw rows exist but display rows are zero due to `minTrades` filter.

## Verification notes
1. Start optimizer loop with `debugOptimizerRows=1`.
   - Confirm loop debug logs show growing raw totals and `appendedCount` even when display-eligible rows are zero.
