1) Console spam removal details
- Removed verbose optimizer row debug logging from `OptimizerPage` for both `[optimizer-rows]` and `[optimizer-rows-append-loop]` paths.
- Kept append-only diagnostics untouched, but row append/update flow no longer emits periodic row spam logs in normal development usage.

2) Paper rearmSec mapping and clamps (why the 400 happened)
- Fixed optimizer row `Copy` mapping so paper `rearmSec` is derived in seconds and never populated with raw milliseconds.
- Added defensive clamp in the copy patch path and config apply path: `paperRearmSec` is clamped to integer `[0..60]`, then converted to `rearmDelayMs = sec * 1000`.
- Added backend normalization in `/api/config` to clamp incoming `paper.rearmDelayMs` to `<= 60000` and drop accidental `paper.rearmSec` from payloads.
- Root cause was unit mismatch risk (ms value flowing into sec field), which could send oversized `rearmDelayMs` and trigger schema 400.

3) Loop seed shifting (loopIndex) and how it prevents repeating the same candidates on each loop
- Added persistent `loopIndex` in optimizer loop state storage.
- `loopIndex` initializes to `0` at loop start and increments after each terminal run in the same loop session.
- Loop starter now passes `loopIndex` into `/api/optimizer/run` payload; optimizer runner computes base random seed as `seed + loopIndex` (and preserves existing per-run offset + blacklist semantics).
- This gives each finite loop iteration a shifted random candidate stream while keeping intra-run dedupe/blacklist behavior intact.

4) How to verify locally
- Build backend and frontend.
- In optimizer UI, start receiving rows and confirm no repeating `[optimizer-rows*]` spam.
- Click `Copy` on optimizer result, verify paper `rearmSec` remains small (`<= 60`), then Apply config and confirm no 400 for `rearmDelayMs`.
- Run finite loop (`runsCount=3`) with fixed seed; verify runs no longer repeat identical candidate stream due only to same seed, and check loop state/progress for incrementing `loopIndex`.
