# Stage 2 - Process Indicators / Observability

## 1) Goal
Added a shared process-status path so UI can show current backend activity for runtime, optimizer, receive-data, and recorder placeholder state.

## 2) Backend changes
1. Added `GET /api/process/status` in `backend/src/api/http.ts`.
2. Endpoint now returns additive process blocks:
   - `runtime`: session state, running start, runtime message
   - `optimizer`: loop state, run index, runs count, current job id, job status, progress percent, message
   - `receiveData`: active job id, state, progress, current symbol, message, eta
   - `recorder`: placeholder state for upcoming recorder stages
3. Added `getActiveReceiveDataJob()` export from `backend/src/dataset/receiveDataStore.ts` for cross-process status aggregation.

## 3) Frontend changes
1. Added `ProcessStatusResponse` typing in `frontend/src/shared/types/domain.ts`.
2. Added status API call `fetchProcessStatus()` in `frontend/src/features/session/api/sessionApi.ts`.
3. Added polling hook `useProcessStatus()` in `frontend/src/features/session/hooks/useProcessStatus.ts`.
4. Added shared UI component `ProcessIndicatorsBar` in `frontend/src/features/session/components/ProcessIndicatorsBar.tsx`.
5. Mounted indicators on:
   - `frontend/src/pages/dashboard/DashboardPage.tsx`
   - `frontend/src/pages/optimizer/OptimizerPage.tsx`

## 4) Tests
1. Added `frontend/src/features/session/components/ProcessIndicatorsBar.test.ts`.
   - validates ETA formatting helper
   - validates percent formatting helper
2. Updated backend receive-data tests to cover active job visibility path assumptions without brittle job-id equality.

## 5) Stability verification
1. `npm --prefix backend test` - passed.
2. `npm --prefix frontend test` - passed.
3. `npm --prefix backend run build` - passed.
4. `npm --prefix frontend run build` - passed.

## 6) Notes
- No breaking contract changes were introduced; endpoint is additive.
- Recorder state is intentionally placeholder until Stage 3 foundation lands.
