1) What changed in runner.ts
- Removed run-index seed shifting for remember-negatives runs. The optimizer now always uses the provided seed directly (`effectiveSeed = baseSeed`) and keeps `seedInfo.runIndex` as `0` for compatibility.
- Replaced the fixed `for` draw loop with an attempts-based `while` loop that keeps evaluating until `args.candidates` non-blacklisted candidates are completed.
- Added blacklist-skip behavior that increments `skippedBlacklisted` and continues without incrementing evaluated count.
- Progress reporting now tracks evaluated candidates (including intra-candidate fractional progress anchored to evaluated count).
- Added a `maxAttempts` guard to prevent infinite loops when many generated candidates are blacklisted. If reached, optimization stops gracefully and includes attempts/evaluated/maxAttempts in diagnostics.

2) Why
- The previous seed behavior (`effectiveSeed = baseSeed + runIndex`) changed RNG output each run, so prior negative parameter sets were not deterministically revisited and skipped. Using the same base seed preserves deterministic candidate generation, allowing blacklist exclusions to reliably prevent repeated negative param sets from reappearing.

3) How to verify locally
- Run one optimization with `rememberNegatives=true` and note negative candidates recorded.
- Run the same optimization inputs again (same tapes/datasets, direction mode, tf, sim, ranges, and seed).
- Confirm that previously negative candidate keys are skipped and not re-evaluated/relisted, while total evaluated candidates still reaches `args.candidates` unless `maxAttempts` is hit.
- Check blacklist summary (`count`, `skipped`) and optional diagnostics (`attempts`, `evaluated`, `maxAttempts`) when saturation occurs.

4) Known limitations
- New candidate parameter sets can still be negative.
- Blacklist behavior only prevents re-evaluating the exact same normalized parameter/simulation key.
