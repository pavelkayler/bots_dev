# 45

## Root cause
- The WebSocket message processing `useEffect` depended on `loopAggState.byRunId` and `minTrades` while also calling `upsertLoopRunRowsAppend`, which mutates loop aggregate state.
- That state update changed `loopAggState.byRunId`, retriggered the effect for the same `lastMsg`, and could repeat until React hit `Maximum update depth exceeded`.

## Fix summary
- Added stable refs (`loopAggByRunIdRef`, `minTradesRef`, `activeJobIdRef`, `loopActiveRef`) so the WS effect reads latest values without depending on mutable state it updates.
- Added `lastProcessedMsgRef` guard so each `lastMsg` string is processed at most once.
- Reduced WS effect dependencies to avoid state-feedback loops.
- Kept snapshot handling, `optimizer_rows_append` routing (loop vs single), and loop hydration behavior unchanged.

## How to verify
1. Start optimizer loop and confirm rows continue to appear and update around 1Hz.
2. Watch browser console while running loop: no `Maximum update depth exceeded` error.
3. Stop loop and confirm no continued loop append spam.
4. For single-job mode, confirm snapshot merge/replace and live row append still update as before.

## Debug flag notes
- `localStorage.debugOptimizerRows=1` still enables optimizer row debug logs.
- Loop append debug log is now throttled (minimum interval) and uses refs, so it does not introduce rerender loops.
