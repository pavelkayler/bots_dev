# Progress Report — Task Batch 2

## Context: what was done previously and why it was wrong
1. In **/progress/1.md** we already implemented loop progress persistence and UI rendering from server-reported progress.
2. Real-world result (your screenshots): loop progress often stayed at **0%** during a run and only became **100%** after the entire loop finished.
3. Root cause found in code (without blaming/overexplaining):
   - The optimizer worker emits two kinds of messages:
     - normal `progress` messages with `donePercent`
     - **blacklist summary** messages that also used `type: "progress"` but **did not include `donePercent`**.
   - Backend treated any `type: "progress"` message as authoritative, so every blacklist summary message effectively set percent to **0** again.
   - UI polling usually observed the reset value, so the bar looked “stuck at 0%”.

## 1) Fix: do not let message-only worker updates reset progress
### Backend
1. Updated the optimizer `onProgress` handler in `backend/src/api/http.ts`:
   - If a worker message has **valid `donePercent`**, update `job.lastPct/job.done` and loop `progress.runPct`.
   - If a worker message has **no `donePercent`** (message-only), **do not touch percent**.
   - Still append `messageAppend` and update `updatedAtMs`.
   - Ensured percent is **monotonic** (never decreases) via `Math.max(job.lastPct, pct2)`.

Why:
- Prevents progress resets to 0 due to blacklist summary messages.

Changed files:
- `backend/src/api/http.ts`

### Worker
2. Improved progress granularity in `backend/src/optimizer/worker/optimizerWorker.ts`:
   - Switched percent computation from integer `Math.floor(...)` to `toDonePercent(done, total)` (up to 2 decimals).
   - Throttled blacklist summary emissions so they don't spam the main thread / disk snapshots.

Why:
- Makes progress smoother and improves ETA stability.

Changed files:
- `backend/src/optimizer/worker/optimizerWorker.ts`

## 2) Fix: elapsed timer must stop after the loop is finished
### Frontend
1. Updated loop polling logic in `frontend/src/pages/optimizer/OptimizerPage.tsx`:
   - When loop becomes **stopped** and `lastJobId` becomes `null`, UI previously stopped polling per-job status and could keep `jobStatus="running"`, so the **Elapsed** timer kept ticking.
   - Now, when loop is not running and not paused and server has a persisted `progress` snapshot, UI derives a **terminal** job status from that snapshot:
     - `done → done`
     - `canceled → cancelled`
     - `error → error`
   - Also sets `jobFinishedAtMs` from `loop.finishedAtMs` or `progress.updatedAt` so elapsed freezes deterministically.

2. Made UI more robust when both sources exist:
   - For an active run, the displayed percent becomes `max(progressSnapshot.runPct, perJobStatusPct)`.

Changed files:
- `frontend/src/pages/optimizer/OptimizerPage.tsx`
