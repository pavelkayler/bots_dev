## 2) Conservative OHLC rule table

| Context | LONG | SHORT | Worst-case tie |
|---|---|---|---|
| Entry fill (limit) | fill if `low <= entryPrice` | fill if `high >= entryPrice` | n/a |
| TP trigger | `high >= tpPrice` | `low <= tpPrice` | if TP and SL both possible, choose SL |
| SL trigger | `low <= slPrice` | `high >= slPrice` | if TP and SL both possible, choose SL |
| Same-bar entry + exit possibility | assume entry fills first; then evaluate SL/TP within same bar | assume entry fills first; then evaluate SL/TP within same bar | choose SL first; if no SL but TP reachable, choose TP; else keep open |

## 3) How to verify locally

1. Baseline behavior unchanged:
   - Run optimizer/paper with default `executionModel` omitted (defaults to `closeOnly`).
   - Compare with prior runs: close-only behavior should remain the same.

2. Conservative mode:
   - Set optimizer run payload `executionModel: "conservativeOhlc"`.
   - Expect more entry fills when 1m low/high touches entry levels.
   - Expect more conservative exits on ambiguous bars because SL wins ties.

3. Automated checks:
   - Run backend tests and builds.
   - Run frontend build.

## 4) Known limitations

- Still bar-based and path-agnostic: no true intra-minute sequencing is simulated.
- Conservative tie-breaking may understate achievable performance in some bars.
- Conservative model only applies when OHLC is available on ticks; otherwise fallback behavior is close-only per-tick checks.
