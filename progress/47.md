# Progress 47

1) Where the limit existed
- Backend schema validation enforced `paper.rearmDelayMs <= 60000` in `backend/src/runtime/configStore.ts`.
- Backend `/api/config` normalization additionally clamped incoming `paper.rearmDelayMs` to `<= 60000` in `backend/src/api/http.ts`.
- Frontend Paper config apply path clamped `paperRearmSec` to `0..60` before converting to milliseconds in `frontend/src/features/config/components/ConfigPanel.tsx`.
- Optimizer → Paper copy flow clamped mapped `rearmSec` to `<= 60` in `frontend/src/pages/optimizer/OptimizerPage.tsx`.

2) What changed
- Removed backend max constraint for `paper.rearmDelayMs`; it is now validated as finite and non-negative only.
- Updated backend `/api/config` normalization to keep integer/non-negative normalization for `paper.rearmDelayMs` without any 60s upper cap.
- Updated frontend config apply logic to preserve unit correctness with `rearmDelayMs = floor/round(rearmSec) * 1000` semantics and no `<= 60` clamp.
- Updated pending optimizer patch application in config panel to map `rearmSec`/`rearmMs` to paper config without upper clamp.
- Updated optimizer copy mapping so `paper.rearmSec` comes directly from `row.params.rearmSec` when present (seconds), otherwise from `row.params.rearmMs / 1000`, with no 60s cap.

3) How to verify locally
- Build backend: `npm --prefix backend run build`.
- Build frontend: `npm --prefix frontend run build`.
- Start app, open Config → Paper, set `rearmSec` to `900` or `3600`, click Apply, verify 200 response and no `invalid_config`.
- Run optimizer with a row that has high rearm value, click Copy, open config panel and verify Paper `rearmSec` matches copied value.
- Apply config, restart app, then GET `/api/config` and verify `paper.rearmDelayMs` persisted as expected large milliseconds value.

4) Notes / risks
- Very large `rearmDelayMs` intentionally reduces entry frequency after closes/exits; this is expected behavior for 1:1 optimizer setting application.
