# 34) Data QA + manifest for Receive Data

1) What changed per file
- `backend/src/dataset/receiveDataStore.ts`
  - Added deterministic QA computation for 1m klines, 5m OI, and funding point series after Receive Data completion.
  - Added duplicate/out-of-order detection, missing-window compression, expected-point coverage calculation, and stable SHA-256 hashes per series.
  - Added manifest writer to `backend/data/cache/manifests/<historyId>.json`.
  - After history row upsert, computes manifest and persists compact manifest summary to dataset history metadata.
- `backend/src/dataset/datasetHistoryStore.ts`
  - Extended `DatasetHistoryRecord` with optional `manifest` summary payload.
  - Added summary normalization during index/meta read.
  - Added `setDatasetHistoryManifestSummary(id, summary)` helper for post-receive updates.
  - Deleting a history row now removes corresponding manifest json file.
- `frontend/src/features/datasetHistory/api/datasetHistoryApi.ts`
  - Extended dataset history API type with optional manifest summary fields for strict typing compatibility.
- `docs/17_optimizer.md`
  - Added “Receive Data QA + manifest” section: validated dimensions, storage path, status semantics, and optimizer trust rationale.

2) Manifest structure and rules
- Full manifest file includes:
  - `historyId`, `universe`, `range`, `generatedAtMs`
  - per-symbol sections: `kline1m`, `oi5m`, `funding`
  - aggregate `summary`
- Compact history summary includes:
  - `status`, `updatedAt`, `coveragePct`
  - missing counts for 1m/OI/funding
  - total duplicates and out-of-order transitions
- Status logic:
  - `ok`: all symbols 100% on 1m + 5m OI and no ordering/dup issues
  - `partial`: aggregate coverage >= 95% but not perfect
  - `bad`: aggregate coverage < 95%, any symbol < 90%, or duplicates/out-of-order found

3) How to verify locally
- Build checks:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- Runtime smoke:
  1. Run backend and trigger `POST /api/data/receive` with short range.
  2. Wait until job status is `done` via `GET /api/data/receive/:jobId`.
  3. Verify manifest file exists at `backend/data/cache/manifests/<jobId>.json`.
  4. Call `GET /api/data/history` and confirm each row has `manifest` summary.
  5. Repeat same target; hashes should remain equal if cache data is unchanged, while `updatedAt` should advance.

4) Known limitations
- Funding is inherently sparse; status rules do not fail solely on funding sparsity.
- Funding expected-point logic uses 8h alignment, matching current receive fetch step.
- Duplicate and out-of-order detection is based on stored JSONL line order; normalized rewrites reduce practical frequency of these anomalies.
