1) What changed per file
- `frontend/src/pages/optimizer/OptimizerPage.tsx`
  - Added two new UI filter toggles in the results toolbar: `val pnl/trade > 0` and `val netPnl > 0`.
  - Added client-side filtering logic so both toggles can be combined with AND semantics, applied after existing `minTrades` and `Hide negative netPnl` display filtering.
  - Added a `Sort: val pnl/trade` quick button that sets `sortKey=valPnlPerTrade` and `sortDir=desc`.
  - Added mild auto-sort behavior: when `val pnl/trade > 0` is enabled while current sort is `netPnl`, it switches to `valPnlPerTrade desc`.
  - Added persisted UI preference keys in localStorage for the new validation filters.
  - Added a safe `valPnlPerTrade` reader fallback (`valNetPnl / valTrades` when direct value is missing, with `valTrades <= 0` treated as 0).
- `docs/17_optimizer.md`
  - Added a Train/Validation note describing the two validation filters and that they are display-only (do not change evaluation scope).
  - Documented the new one-click validation sort preset.
- `progress/38.md`
  - Added this stage report.

2) Why
- Surface more stable candidates faster during operator triage.
- Reduce overfit chasing by making positive validation performance easy to isolate and rank.
- Keep behavior safe and minimal by constraining changes to UI display filtering/sorting only.

3) How to verify locally
- Build frontend:
  - `npm --prefix frontend run build`
- Start app/backend as in your normal local workflow and run optimizer until result rows include mixed validation outcomes.
- In Optimizer results table:
  - Enable `val pnl/trade > 0` and confirm rows with `valPnlPerTrade <= 0` disappear.
  - Enable `val netPnl > 0` and confirm rows with `valNetPnl <= 0` disappear.
  - Enable both and confirm only rows passing both conditions remain.
  - Click `Sort: val pnl/trade` and confirm descending ordering by validation pnl/trade.
- Edge safety:
  - Confirm no crash or NaN UI behavior when `valTrades == 0`; fallback treats validation pnl/trade as 0.

4) Known limitations
- Filters are display-only in the frontend; backend evaluation still runs all candidates.
- Positive validation metrics can still be noisy on small sample sizes (especially very low `valTrades`).
