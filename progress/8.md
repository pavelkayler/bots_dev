# 8 — Optimizer results table: stop full remount on each new row

## Problem
While optimizer results started arriving “smoothly”, each time a new result arrived the UI visually re-rendered the entire results table from 0 to N again (as if the list was rebuilt), instead of appending one new row.

## Root cause
The `<tr>` key used for result rows was unstable:
- it included the **array index** `i` and **changing fields** (`netPnl`, `trades`),
- so on each update React treated *all* existing rows as new elements and remounted them.

This causes visible “restart from 0” rendering/animations even if the data append is correct.

## Fix
Use a stable key per result row:
- `makeResultSignature(row)` is used as the base stable identifier (params signature),
- additionally, if `__runJobId` exists it is prefixed to avoid collisions between different runs.

## Files changed
- `frontend/src/pages/optimizer/OptimizerPage.tsx`

## Verification
1) Start the app (`m.bat stop` → `m.bat start`).
2) Open `/optimizer`.
3) Start a run/loop.
4) Observe the results table:
   - existing rows do **not** remount/re-animate on each new incoming row,
   - only the new row appears/updates, the table does not “rebuild from 0”.
