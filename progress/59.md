# 59 — Multi-bot architecture split (phase 1, current bot only)

## 1) Why this split was needed
1. Runtime and optimizer were implicitly single-bot and tightly coupled to one monolithic config.
2. Strategy semantics (including TP/SL) and generic execution/session policy were mixed in one `paper` block.
3. Presets were single bucket objects, making future multi-bot extension hard and unsafe.

## 2) What moved into bot config
1. Added bot registry boundary: `backend/src/bots/registry.ts` with current bot `oi-momentum-v1`.
2. Introduced `botConfig` in runtime config state:
   - `fundingCooldown`
   - `signals`
   - `strategy`:
     - `klineTfMin`
     - `entryOffsetPct`
     - `entryTimeoutSec`
     - `tpRoiPct`
     - `slRoiPct`
     - `rearmDelayMs`
     - `applyFunding`
3. TP/SL explicitly remains in `botConfig.strategy` only.

## 3) What stayed in execution profile
1. Introduced `executionProfile` in runtime config state:
   - execution mode
   - direction gate
   - margin/leverage
   - maker fee
   - max daily loss
   - risk limits block
2. No TP/SL in execution profile.

## 4) selectedBotId introduction points
1. Runtime config now contains:
   - `selectedBotId`
   - `selectedBotPresetId`
   - `selectedExecutionProfileId`
2. Added API:
   - `GET /api/bots`
   - `GET /api/config/selections`
   - `POST /api/config/selections`
3. Optimizer run/loop payloads now carry:
   - `selectedBotId`
   - `selectedBotPresetId`
4. Optimizer history run payload keeps bot selection metadata.

## 5) Storage split: bot presets vs execution profiles
1. Added bot preset store:
   - `backend/src/presets/botPresetStore.ts`
   - files in `backend/data/bot_presets/`
2. Added execution profile store:
   - `backend/src/presets/executionProfileStore.ts`
   - files in `backend/data/execution_profiles/`
3. Added APIs:
   - `GET/PUT/DELETE /api/bot-presets*`
   - `GET/PUT/DELETE /api/execution-profiles*`
4. Legacy presets/config remain readable for migration/backward compatibility.

## 6) Runtime config migration and resolution changes
1. Reworked `backend/src/runtime/configStore.ts`:
   - migrates old monolithic config into split state
   - keeps resolved compatibility fields (`signals`, `fundingCooldown`, `paper`, etc.) for existing runtime/optimizer consumers
2. Added selection apply logic before session start and optimizer run/loop.

## 7) UI/page changes (minimal)
1. Added bot page:
   - `frontend/src/pages/bots/BotsPage.tsx`
2. Added shared selector component:
   - `frontend/src/features/bots/components/BotExecutionSelectors.tsx`
   - `frontend/src/features/bots/hooks/useBotSelections.ts`
   - `frontend/src/features/bots/api.ts`
3. Dashboard now focuses on operator execution controls:
   - added `ExecutionPanel`
   - moved bot/profile selection into dedicated selector card
4. Optimizer is bot-aware:
   - selector shown on optimizer page
   - selected bot/preset sent in loop payload
5. Navigation includes `/bots`.
