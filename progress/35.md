# 35) Optimizer decision cadence policy (Stage 2)

1) What changed
- `backend/src/optimizer/runner.ts`
  - Replaced per-minute/bucket-based signal reference fallback logic with a per-symbol signal-window cadence state:
    - `prevWindowClose`
    - `prevWindowOivClose`
  - Kept market cache updates on every ticker event.
  - Kept `paper.tick(...)` on every ticker event so fills/TP/SL/expiry continue to be processed each minute.
  - Added window-close gate for decisioning using `isWindowClose = (ts % tfMs) === 0`.
  - Signal engine is now called only on window closes.
  - On non-window ticks, passes `signal=null` and `signalReason="wait_window_close"`.
  - On first window close per symbol (missing refs), seeds refs from current close/OIV and skips decision (`signal=null`, `signalReason="window_seed"`).
  - Computes `priceMovePct` and `oivMovePct` from previous signal-window close refs (not bucket rollover state).
  - Decision counters (`decisionsNoRefs`, `signalsOk`) are updated only when a decision attempt actually runs.
- `docs/17_optimizer.md`
  - Updated replay/decision mechanics to explicitly document:
    - close-only execution ticks
    - decisioning only at `tf(opt)` window closes
    - execution-only behavior between windows
    - reference deltas measured between consecutive signal-window closes
  - Added rationale section clarifying why optimizer is not “better than paper”.

2) Why
- Prevents over-frequent per-minute entry decisions in replay.
- Aligns optimizer with the Stage 2 policy: one decision attempt per `tf(opt)` boundary per symbol.
- Ensures reference values are semantically stable and window-aligned, removing bucket rollover ref drift.

3) How to verify locally
1. Build:
   - `npm --prefix backend run build`
   - `npm --prefix frontend run build`
2. Run optimizer with `tf(opt)=15` (or 30/60).
3. Confirm behavior in logs/checkpoints:
   - `signalsOk` / decision attempts increment roughly once per 15-minute window per symbol (not every minute).
   - fill/TP/SL/expiry events still occur between window closes, proving `paper.tick` still runs per minute.
4. Compare with previous run:
   - expect fewer entries/trades due to reduced decision cadence
   - execution lifecycle counters remain active between window boundaries.

4) Known limitations
- Replay remains close-only (no OHLC intrabar extrema).
- Funding history is applied as last-known step series from cached funding points.
- Decision generation is intentionally restricted to window-boundary timestamps only.
