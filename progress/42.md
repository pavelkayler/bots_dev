1) What changed per file
- `backend/src/optimizer/runner.ts`
  - Added `rowId` (required) and `candidateKey` (optional) to `OptimizerResult`.
  - Added optional `jobId` / `runId` to `RunOptimizationArgs` so row identity can be built in runner.
  - When each candidate result row is created, now sets:
    - `candidateKey` via the existing deterministic `buildCandidateKey(...)`.
    - `rowId` as `${jobId}:${runId}:${candidateKey}` (with safe fallbacks).
  - Because streamed append and final results reuse `OptimizerResult`, both payload paths now include `rowId`.

- `backend/src/api/http.ts`
  - Updated result merge keying to prefer `row.rowId` when present, with legacy signature fallback.
  - Added `jobId` and `runId` into worker run payload so backend row IDs are stable for the job.

- `frontend/src/features/optimizer/api/optimizerApi.ts`
  - Extended `OptimizationResult` with `rowId: string` and optional `candidateKey?: string`.

- `frontend/src/pages/optimizer/OptimizerPage.tsx`
  - Added `resolveRowId(row)` helper and changed merge/dedup logic to key rows by `rowId` (fallback to legacy signature only if missing).
  - Updated single-run and loop aggregation paths (`snapshot`, `rows_append`, polling hydration, final page fetches) to upsert by `rowId` using `rowsById` maps and stable `order` arrays.
  - Updated render keys/debug tracking to align with `rowId`.
  - Kept sorting/pagination operating on derived rows from `order -> rowsById`.
  - On loop start reset, also clears pending live append cache and loop draft storage reset remains in place.

2) Why
- Duplicates came from unstable/insufficient dedup keys during incremental merges (especially when backend re-sent top rows or snapshots). Stable `rowId` ensures repeated sends replace the same record instead of creating additional rows.

3) How to verify
- Build checks:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- Runtime verification:
  - Start optimizer loop with `runsCount=3`, `candidates=200`.
  - While running, confirm repeated streamed rows update existing entries and do not increase count incorrectly.
  - After completion, confirm no duplicated identical rows in results table.

4) Known limitations
- Existing old persisted drafts/snapshots without `rowId` still rely on fallback signature keying until refreshed by new backend payloads.
- `runId` currently mirrors job ID in the run payload; uniqueness/stability requirement is still satisfied because each run has a unique job ID and candidateKey is deterministic.
