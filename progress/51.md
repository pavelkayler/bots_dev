# 51 — Stability hardening set (Hard Stop, anti-stacking, shared execution rules)

## 1) Problem statements
1. STOP requests could wait on in-flight async work and leave runtime in `STOPPING` longer than expected.
2. Entry signals could repeatedly attempt orders unless symbol state was explicitly enforced across the executor lifecycle.
3. TP/SL and limit-fill logic needed a single shared implementation so paper execution and optimizer replay stay consistent.

## 2) Root causes
1. Runtime stop/start flow had no per-run cancellation primitive owned by session runtime; cleanup paths awaited remote calls directly.
2. Broker paths tracked order/position booleans but did not expose a strict symbol execution-state invariant in logs/guards.
3. Fill logic lived inline in paper broker; no shared module documented as the canonical rule implementation.

## 3) Changes
1. Added runtime run context in `backend/src/runtime/runtime.ts` with `runId`, `AbortController`, `stopRequestedAt`, and idempotent `stopPromise`.
2. STOP now aborts run context immediately and races stop completion against a hard overall timeout; stop finalization always forces `STOPPED` and clears run context in `finally`.
3. Wrapped demo wallet-balance awaits in timeout + abort handling for startup/stop paths.
4. Added transition helper logging fromState->toState with runId payload.
5. Added shared module `backend/src/execution/executionRules.ts` containing:
   - `evaluateLimitFill(...)`
   - `evaluateTpSl(...)` with deterministic worst-case tie-break for overlap candles.
6. Wired paper broker to use the shared execution rules for limit fills and TP/SL checks. Optimizer replay remains aligned because it executes through paper broker.
7. Added symbol execution state machine (`FLAT|OPENING|OPEN|CLOSING`) and entry-attempt counters in paper/demo brokers, with entry-skip guard when symbol is not flat.
8. Added deterministic entry IDs per run+symbol+attempt in paper/demo brokers for idempotent OPENING behavior.

## 4) Behavioral timeline
1. Start: runtime creates run context (`runId`, `AbortController`) and enters `RESUMING`.
2. Ready: waitForReady uses run-context signal; if aborted, start exits to `STOPPED`.
3. Running: per-symbol entry allowed only from `FLAT`.
4. Stop requested: runtime enters `STOPPING`, sets `stopRequestedAt`, aborts signal immediately, performs best-effort cleanup with bounded waits.
5. Finalize: runtime always transitions to `STOPPED` in finalization path; repeated STOP calls share same stop promise while active.

## 5) Verification steps
1. Backend tests include execution-rules unit coverage for conservative limit fill and TP/SL tie-break behavior.
2. Backend build validates strict TS for runtime/broker changes.
3. Frontend build validates no contract-breaking compile issues from API wiring changes.
