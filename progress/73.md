# 73 - Signal bot integration (runtime + optimizer + UI selection)

Date: 2026-03-07

## 1) Goal
Integrate one additional signal bot into the existing foundations so it can be selected and launched from current operational pages:
- Dashboard runtime flow
- Optimizer historical replay flow

No separate execution engine was introduced. Existing runtime and optimizer engines were reused.

## 2) Backend changes

### 2.1 Bot registry
File:
- `backend/src/bots/registry.ts`

Added a new bot entry:
- `signal-multi-factor-v1` (`Signal Multi-Factor`)

Registry now exposes:
- `oi-momentum-v1`
- `signal-multi-factor-v1`

Default config normalization/validation remains shared and backward-compatible.

### 2.2 Signal model implementation
File:
- `backend/src/engine/SignalEngine.ts`

Added model switch in signal config:
- `model: "oi-momentum-v1" | "signal-multi-factor-v1"`

Behavior:
- `oi-momentum-v1`: existing threshold logic
- `signal-multi-factor-v1`: weighted score from:
  - normalized price move
  - normalized OI move
  - funding contribution
  - direction gate and funding-sign policy remain enforced

### 2.3 Runtime wiring
File:
- `backend/src/api/wsHub.ts`

Runtime signal engine now initializes model by selected bot id:
- selected `signal-multi-factor-v1` -> multi-factor model
- otherwise -> momentum model

### 2.4 Optimizer wiring
File:
- `backend/src/optimizer/runner.ts`

Added `selectedBotId?: string` to run args and routed model choice into replay decision engine.
Optimizer now evaluates candidate runs using the selected bot model semantics.

## 3) Frontend changes

### 3.1 New page
Files:
- `frontend/src/pages/signalBot/SignalBotPage.tsx`
- `frontend/src/app/routing/routes.tsx`
- `frontend/src/pages/dashboard/components/HeaderBar.tsx`

Added `/signal-bot` route and nav item.
Page includes:
- bot selection controls
- process indicators
- explicit operational note: runtime from Dashboard, historical replay from Optimizer
- data policy note: Bybit primary, CoinGlass optional and rate-limited

Bot selection on Dashboard and Optimizer remained the primary control path.

### 3.2 Type contract alignment
Already aligned in prior fixes and kept stable.

## 4) Tests added/updated

Added:
- `backend/src/tests/botRegistry.test.ts`
  - verifies both bot ids are exposed
  - verifies signal bot defaults resolve

Updated:
- `backend/src/tests/signalEngine.test.ts`
  - added coverage for multi-factor model long/short and funding mismatch behavior

## 5) Data-source / plan constraint handling
- Runtime and optimizer are still Bybit-first and operational without CoinGlass.
- CoinGlass remains optional enrichment, not startup-critical.
- Hobbyist constraints are explicitly documented (30 req/min baseline, 1m refresh cadence).
- Higher-tier metric availability is documented as plan-dependent and must be validated at integration time.
