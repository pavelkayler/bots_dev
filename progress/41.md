1) What changed per file
- `backend/src/paper/PaperBroker.ts`: added optional `executionModel` (`closeOnly` default, `conservativeOhlc` optional), optional tick-level `ohlc`, conservative range-based fill/exit checks, and same-bar entry+exit pessimistic handling (SL preferred when both possible) while preserving current event/PnL semantics.
- `backend/src/optimizer/runner.ts`: replay now forwards per-candle OHLC data on the same single close tick, and optimizer run args include `executionModel` so candidates can run in either model without changing decision cadence.
- `backend/src/api/http.ts`: optimizer run/loop payload parsing and validation accept `executionModel` limited to `closeOnly|conservativeOhlc`.
- `frontend/src/features/optimizer/api/optimizerApi.ts`: added frontend payload typing for `executionModel`.
- `frontend/src/pages/optimizer/OptimizerPage.tsx`: added a minimal execution-mode selector (`Close-only (safe)` / `Conservative OHLC`), persisted in localStorage, and sent in optimizer loop payload.
- `docs/04_paper_execution.md`, `docs/17_optimizer.md`: documented close-only vs conservative-OHLC semantics and explicit SL-first tie-break for TP+SL-in-range bars.

2) Conservative OHLC rule table
- Pending LONG entry fill: `low <= entryPrice`
- Pending SHORT entry fill: `high >= entryPrice`
- Open LONG TP hit: `high >= tpPrice`
- Open LONG SL hit: `low <= slPrice`
- Open SHORT TP hit: `low <= tpPrice`
- Open SHORT SL hit: `high >= slPrice`
- TP and SL both hit in same bar (long/short): close by SL (worst case)
- Entry fills and TP/SL also hit in same bar: apply SL first if reachable, else TP, else keep open
- `closeOnly` or missing OHLC: retain existing mark-price-based behavior

3) How to verify locally (including A/B comparison)
- Build:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- Optional unit check:
  - `npm --prefix backend run test -- paperBroker`
- A/B execution comparison on same dataset + same seed:
  1. Run optimizer with `executionModel=closeOnly` and export trades for a selected candidate.
  2. Run optimizer with `executionModel=conservativeOhlc` using the same candidate count, seed, tf, direction, and dataset.
  3. Compare counters and trades:
     - conservative mode can fill more pending limits due to range touches,
     - bars where TP and SL are both reachable close by SL,
     - conservative mode should not systematically beat close-only over identical runs.

4) Known limitations
- Execution remains bar-based (1m range checks only), with no intra-bar path simulation.
- Conservative tie-breaking is intentionally pessimistic and may understate achievable live performance in some markets.
- Fill/close prices remain at configured levels (entry/tp/sl), not bar extremum or VWAP.
