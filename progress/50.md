# 50 — DEMO Bybit signing drift and leverage fallback fixes

1) What happened
- DEMO signed REST calls could fail with `retCode=10002` (`invalid request, please check your server timestamp or recv_window`) when local clock drift plus network delay exceeded the active recv window.
- DEMO leverage setup could fail with `retCode=10001` / `buy leverage invalid` for symbols with low max leverage like `SIRENUSDT`, and the broker still marked leverage as configured.

2) Root cause in code
- Signing timestamp source used raw `Date.now()` directly at request time without exchange time offset compensation.
- Signed requests defaulted `BYBIT_RECV_WINDOW` fallback to `5000` ms.
- In `DemoBroker`, leverage setup used `finally` to set `leverageSet`, so failures were treated as success and never retried.

3) Exact changes
- Added `backend/src/bybit/serverTimeOffset.ts`:
  - `refreshServerTimeOffset(client)` fetches `/v5/market/time`, computes process offset (`serverTimeMs - midpoint(localBefore, localAfter)`), and preserves previous offset on failures.
  - `nowMs()` returns `Date.now() + offsetMs`.
  - `startServerTimeOffsetScheduler(...)` triggers startup refresh and periodic refresh every 5 minutes.
- Updated `backend/src/bybit/v5Auth.ts`:
  - `buildSignedHeaders` now defaults timestamp to `nowMs()` when caller omits one.
- Updated `backend/src/bybit/BybitDemoRestClient.ts`:
  - Default recv window fallback increased from `5000` to `20000` while still honoring `BYBIT_RECV_WINDOW` env override.
  - Starts server time offset scheduler in constructor.
  - Uses `nowMs()` for `X-BAPI-TIMESTAMP` during signing.
- Updated `backend/src/demo/DemoBroker.ts` leverage flow:
  - Added leverage invalid detection (`retCode=10001` and/or `retMsg` contains `leverage invalid`).
  - Added one retry path with fallback clamp leverage from symbol max.
  - Added minimal per-symbol fallback map with `SIRENUSDT: 5`.
  - Added optional one-time instrument max leverage lookup cache via `getInstrumentsInfoLinear({symbol})`.
  - `leverageSet` is now recorded only after successful leverage set.
  - If both attempts fail, logs a clear stage (`setLeverage` / `setLeverageFallback`) and exits tick safely.

4) Verification steps
- Build checks:
  - `cd backend && npm run build`
  - `cd frontend && npm run build`
- Runtime checks for DEMO mode:
  - Start backend in DEMO mode and verify logs do not repeatedly emit `retCode=10002`.
  - Observe startup/periodic server offset refresh behavior.
  - For `SIRENUSDT` with configured leverage above max, verify first leverage attempt fails and one fallback attempt with leverage 5 succeeds, then order flow continues.
