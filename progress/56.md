## 1) Original weaknesses confirmed
1. `maxTradesPerDay` was consumed by placement events (`ORDER_PLACED`, `DEMO_ORDER_PLACE`) instead of actual opened entries.
2. Emergency stop reason/message existed, but sticky lifecycle semantics were incomplete:
   - no dedicated persistent emergency-stop flag/reason state
   - non-running trigger paths could avoid a full stop call and still needed a hard block against continuation/resume.
3. Focused tests for runtime risk-limit semantics were missing.

## 2) Runtime accounting changes
1. Updated risk event accounting in [runtime.ts](/C:/Users/foxen/Desktop/bots_dev/backend/src/runtime/runtime.ts):
   - `riskEntriesPerDay` now increments only on actual open/fill events:
     - `ORDER_FILLED`
     - `POSITION_OPEN`
     - `DEMO_POSITION_OPEN`
2. Removed daily-trade increments on placement attempts:
   - `ORDER_PLACED` and `DEMO_ORDER_PLACE` no longer consume daily trade budget.
3. Kept consecutive-error reset tied to successful actual entry open/fill, not placement attempts.

## 3) Emergency-stop sticky lifecycle changes
1. Added sticky runtime state in [runtime.ts](/C:/Users/foxen/Desktop/bots_dev/backend/src/runtime/runtime.ts):
   - `emergencyStopActive: boolean`
   - `emergencyStopReason: string | null`
2. On breach (`maybeTriggerEmergencyStop`):
   - sets sticky state and reason once
   - keeps `runtimeMessage` aligned: `Emergency stop: <reason>`
   - logs `EMERGENCY_STOP`
   - triggers hardened `stop()` when state is `RUNNING`/`RESUMING` (existing behavior retained).
3. Added trading/resume guards:
   - `shouldAllowEntry` returns `false` while emergency stop is active.
   - `resume()` in `PAUSED` now refuses to resume while emergency stop is active and preserves the reason message.
4. Sticky state clear point:
   - sticky emergency-stop state is not cleared during stop.
   - it is cleared on clean start-cycle reset (`start()` -> `resetRiskState(...)`).
