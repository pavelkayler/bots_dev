1) What was wrong
- Loop results were being dropped in the WebSocket append handler because loop ingestion was gated by `loopActive && jobId === loopJobId`.
- In real runs this fails frequently: `loopJobId` can be null early, WS `jobId` is a run id (not guaranteed to match any loop controller id), and loop display can already be active while `loopActive` is still false.

2) What changed
- Updated loop WS ingestion invariant in `OptimizerPage.tsx`: when `lastTableSourceRef.current === "loop"`, every `optimizer_rows_append` batch is accepted and appended by `payload.jobId` via `upsertLoopRunRowsAppend(jobId, rows, "ws")`.
- Kept single-run behavior intact outside loop display: only append batches for `activeJobId`.
- Reworked hydration to run by current run id from loop status (`loopStatus?.loop?.lastJobId`) and only when that run has no stored rows.
- Hydration is once-per-run-id and append-merge only; it does not clear or rebuild loop aggregate state.

3) How to verify
- Start optimizer loop and open results table.
- Confirm first WS append batch appears immediately in loop results without refresh and without waiting for loop status polling.
- Enable debug logs in devtools console:
  - `localStorage.setItem("debugOptimizerRows", "1")`
  - Observe logs: `[optimizer-rows-append-loop] { jobId, batchSize, totalBefore, totalAfter }`
- Optional refresh test: if loop table is empty after refresh, hydration fetches by `loopStatus.loop.lastJobId` once and merges rows into loop aggregate.

4) Why this prevents recurrence
- Explicit invariant is now encoded in code path: loop display must never drop `optimizer_rows_append` messages.
- Loop ingest no longer depends on transient `loopActive`/`loopJobId` matching state.
- Poll hydration is constrained to a safe fallback (current run id + empty store + once per run), preventing overwrite/clear behavior.
