# Progress Report 14

## Root cause
Per-run replay was caused by replace-style updates for the active loop run in two places:
- periodic loop polling fetched all rows and replaced the current run slice
- WebSocket append handling rebuilt the current run map/array each batch

Both paths recreated row objects/arrays for already-rendered rows in the active run, producing visible 0..N replay while the run was executing.

## Files changed
- `frontend/src/pages/optimizer/OptimizerPage.tsx`

## What changed
- Replaced loop aggregate array state with per-run append-only store:
  - `LoopAggState` with `runOrder` and `byRunId`
  - each run uses `rowsById` + `order` for stable row identity
- Added derived render list (`loopAggRowsForRender`) from store order/maps without spreading existing row objects.
- Replaced loop live update path with merge-only helper `upsertLoopRunRowsAppend(jobId, rows, source)`:
  - appends new signatures only
  - keeps existing object references untouched
  - updates totals/version only when changed
- Updated completed-run hydration to use merge-only helper (no replace behavior).
- Removed active-run periodic replace polling behavior:
  - now does one hydration fetch only when active run store is empty (refresh recovery)
  - otherwise relies on WS append flow
- Kept loop draft localStorage compatibility by serializing derived row list array.
- Loop-mode row stability:
  - loop rendering no longer passes `rowIndex` to row component
  - key remains stable: `${__runJobId}:${signature}`
- DEV instrumentation (`localStorage.debugRunAppendOnly === "1"`):
  - logs append source (`ws` / `poll` hydrate)
  - logs loop store replacement detection if active-run total decreases
  - row mount/unmount tracer logs for first two loop rows and newest tracked row signatures
