# Progress Report 52

## 1) Problem
1. `r and b.bat` had extra operational logic (cache preserve/restore, git add/commit/push, interactive menu) that is outside the required restart scope.
2. Live row updates under heavy upstream traffic needed coalescing and rate limiting so frontend receives fewer, steadier WS updates.

## 2) Root cause
1. The batch script accumulated non-essential workflows over time.
2. WS live updates were not buffered by event key before flush, so high incoming ticker/kline volume could trigger excessive update churn and UI jitter.

## 3) Fix implemented
1. Simplified `r and b.bat` to exactly three sequential steps with error propagation and pause behavior:
   - `replace.bat`
   - `m.bat install`
   - `m.bat start`
2. Added backend live update coalescing utility:
   - `backend/src/api/liveUpdateAggregator.ts`
   - key-based `Map` coalescing
   - fixed 100ms flush interval (up to 10Hz)
   - bounded key count (`maxKeys=5000`) with one warning on overflow.
3. Integrated aggregator into WS hub (`backend/src/api/wsHub.ts`):
   - enqueue per incoming ticker (`ticker:<symbol>`)
   - enqueue per confirmed kline (`kline:<symbol>`)
   - on flush, send one compatible `tick` payload (same schema)
   - lifecycle-safe: start only in `RUNNING`; stop+clear on stop/pause/close.
4. Updated docs to match coalesced cadence:
   - `docs/06_contracts.md`
   - `docs/11_current_state.md`
   - `docs/12_live_rows.md`
