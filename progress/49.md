# 49 — Fix #5 (Priority A.1): RUNNING + uptime only after Streams ON

## 1) Exact problem
When the operator clicked **Start**, backend session state could become `RUNNING` before the Bybit public websocket streams were actually connected. This made the UI show `RUNNING` and start uptime too early, while `Streams` could still be connecting.

## 2) Root cause in current code
1. `runtime.start()` set `sessionState` to `RUNNING` immediately after `RESUMING`, without waiting for stream readiness.
2. `wsHub` only initiated upstream connect when runtime state was already `RUNNING`, so stream connect was downstream of session running.
3. Uptime in the dashboard was derived from session id timestamp, which is created before actual stream readiness.

## 3) What was changed

### Backend runtime/session start sequencing
- **File:** `backend/src/runtime/runtime.ts`
1. Added `runningSinceMs` to runtime status.
2. Extended `start()` to accept `waitForReady` async hook.
3. `start()` now:
   - emits `RESUMING`,
   - awaits readiness hook,
   - only then switches to `RUNNING` and sets `runningSinceMs`.
4. On readiness failure/cancel, start exits safely to `STOPPED`, resets running timestamps, and does not enter `RUNNING`.

### WS hub readiness primitive + event-driven wait
- **File:** `backend/src/api/wsHub.ts`
1. Added exported readiness API:
   - `awaitAllStreamsConnected({ timeoutMs, signal })`
2. Added internal waiter set with timeout + abort handling.
3. Waiters resolve on `onOpen` (`streamsEnabled && bybitConnected`), reject on close/error/stop/timeout/abort.
4. Updated runtime stream lifecycle sync to keep streams active for `RESUMING` (connect-before-running) and start tick timer only in `RUNNING`.

### Session start/stop HTTP orchestration
- **File:** `backend/src/api/http.ts`
1. `POST /api/session/start` now creates an `AbortController` and calls:
   - `runtime.start({ waitForReady: () => awaitAllStreamsConnected(...) })`
2. Timeout is 20s.
3. If readiness fails, endpoint returns `503` with `streams_not_ready` instead of reporting a running session.
4. `POST /api/session/stop` aborts any in-flight start wait before calling `runtime.stop()`.

### Frontend uptime correctness
- **Files:**
  - `frontend/src/shared/types/domain.ts`
  - `frontend/src/features/ws/hooks/useWsFeed.ts`
  - `frontend/src/features/session/hooks/useSessionRuntime.ts`
  - `frontend/src/pages/dashboard/DashboardPage.tsx`
1. Added backward-compatible `runningSinceMs` to session/websocket snapshot typing.
2. WS feed now stores `wsRunningSinceMs` from snapshots.
3. Dashboard uptime source is now:
   - `runningSinceMs` (WS/status), fallback to old session-id parsing.
4. Result: uptime starts only when backend reports `RUNNING` after stream readiness.

### Docs
- Updated contracts/current-state docs:
  - `docs/06_contracts.md`
  - `docs/11_current_state.md`

## 4) New behavior timeline (state transitions)
1. User clicks Start.
2. Backend sets session `RESUMING`.
3. WS hub starts connecting required streams.
4. If all required streams connect within timeout:
   - backend sets `RUNNING`,
   - sets `runningSinceMs`,
   - UI RUNNING badge + uptime begin.
5. If connect fails/times out/stop requested while waiting:
   - session never reaches `RUNNING`,
   - uptime does not start,
   - start endpoint returns an error (`streams_not_ready`) and session remains non-running.

## 5) How to verify locally
1. Start backend + frontend.
2. Click Start with valid universe selected.
3. Observe sequence:
   - Session shows `RESUMING`,
   - Streams connect,
   - only then Session becomes `RUNNING` and uptime starts.
4. Simulate upstream failure (e.g. invalid ws URL/network block), click Start:
   - Session does not become `RUNNING`,
   - uptime remains stopped,
   - start call returns readiness error.
5. During `RESUMING`, click Stop:
   - in-flight start is canceled,
   - session remains non-running, no stuck starting state.
