1) What changed per file
- `backend/src/dataset/receiveDataStore.ts`
  - Added always-on interval ensure logic for Receive Data (`1m` + selected interval) and switched OI retrieval to fixed `5min`.
  - Added incremental missing-window detection for klines and OI, so only absent ranges are requested.
  - Added dedicated OI cache IO (`data/cache/bybit_open_interest/5min/<symbol>.jsonl`) with load/merge/write.
  - Added post-fetch OI re-alignment over all in-range kline rows (for both 1m and selected interval caches), updating existing rows too.
  - Updated job progress accounting so total/completed steps reflect executed kline fetch windows only.
- `frontend/src/features/datasetTarget/ui/DatasetTargetCard.tsx`
  - Removed `1` and `3` timeframe options from dataset target UI.
  - Changed draft default interval to `15` and ensured persisted invalid/legacy values coerce to `15`.
  - Slowed receive-job polling interval to `1000ms`.
- `backend/src/dataset/datasetTargetStore.ts`
  - Changed default dataset target interval from `1` to `15` (interval union remains unchanged).
- `backend/src/optimizer/runner.ts`
  - Cache replay now prefers `1m` cache per symbol, with fallback to selected dataset interval if `1m` file is missing.
  - For `1m` replay, emits one close-only ticker per minute candle (no synthetic open/high/low points, no `kline_confirm` emissions in this path).
  - Added `directionMode` to optimizer result rows.
- `backend/src/api/http.ts`
  - Added dataset validation requiring both selected interval cache and `1m` cache for each symbol in optimizer run/loop start.
  - Tightened optimizer timeframe validation to reject `optTfMin < 5` for run and loop start.
- `frontend/src/features/optimizer/api/optimizerApi.ts`
  - Extended `OptimizationResult` type with optional `directionMode`.
- `frontend/src/pages/optimizer/OptimizerPage.tsx`
  - Removed `1` and `3` from optimizer TF selector.
  - Default optimizer TF changed to `15`; persisted invalid/legacy values are coerced to `15`; storage writes now only keep values `>=5`.
  - Added `direction` column to the main results table and history expanded results table; adjusted empty-state `colSpan`.
  - Reduced polling/request frequency:
    - active results poll `800 -> 1000ms`
    - dataset-cache sync `500 -> 1000ms`
    - running clock interval `500 -> 1000ms`
    - single-job status throttle `200 -> 1000ms`
    - stabilized loop-status polling dependency to `loopStatus?.loop?.loopId`.

2) Why (map to tasks A–E)
- A (Receive Data + defaults)
  - Implemented UI/default interval changes and backend default interval shift to `15`.
  - Receive Data now always ensures `1m` + selected kline caches and OI `5min` coverage for requested ranges.
  - Added incremental fetch windows and OI cache persistence to avoid re-downloading covered data.
  - Re-alignment step updates OI on all in-range rows, including existing cache rows.
  - Progress now tracks only actual kline fetch windows.
- B (optimizer replay fidelity)
  - Cache replay now prefers `1m` data and uses close-only minute ticks, with lightweight event volume.
  - Fallback remains compatible with old interval-only caches.
  - API now requires `1m` cache presence to run loop/single optimizer jobs.
- C (optimizer TF restrictions)
  - UI no longer offers minute TF below 5 and defaults to `15`.
  - Backend enforces `optTfMin >= 5`.
- D (direction in results)
  - Added direction mode to backend result rows and rendered it in both primary and history-expanded results tables.
- E (poll/request rate control)
  - Increased polling/throttle intervals and fixed unstable effect dependency that caused repeated refresh scheduling.

3) How to verify locally (commands + UI steps)
- Build checks:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- Receive Data behavior:
  1. Open `/optimizer`, set Dataset Target interval to `15` (or any available TF), start Receive Data.
  2. Confirm cache files exist after completion:
     - `backend/data/cache/bybit_klines/1/<symbol>.jsonl`
     - `backend/data/cache/bybit_klines/<selected>/<symbol>.jsonl`
     - `backend/data/cache/bybit_open_interest/5min/<symbol>.jsonl`
  3. Re-run Receive Data for overlapping range and verify job completes quickly with few/no kline progress steps.
- Optimizer replay + validation:
  1. Start optimizer run from dataset history selection.
  2. Confirm run starts only when both selected interval and `1m` caches exist.
  3. Verify results table has `direction` column and values (`both/long/short`).
  4. Expand a history row and verify its nested results table also shows `direction`.
- Polling frequency:
  1. Keep optimizer page open with loop status visible.
  2. In browser devtools Network tab, verify polling calls are not faster than about once per second.

4) Known limitations intentionally kept (minimal diff)
- Kept existing dataset history interval selection behavior; history still records selected interval while replay path prefers `1m` internally.
- Did not add new API schema versioning for result rows; `directionMode` is additive.
- OI/kline missing-window logic assumes grid-aligned expected timestamps and does not attempt gap interpolation beyond requested range.
