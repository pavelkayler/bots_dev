1. Replaced the queue-based limiter implementation with a fixed-window limiter in `receiveDataStore`:

2. Set exact constants:

3. Kept request batch span logic separate (`KLINE_BATCH_SPAN_MS`) so API paging behavior is unchanged.

Why:
- Requirement D asked for exact 500/5s limit with no additional dependencies.

## 4) Optimizer loop progress: persisted backend state + server-reported frontend rendering
1. Backend loop state model now includes persisted `progress` snapshot with fields:
   - `jobId`
   - `status` (`running | done | canceled | error`)
   - `runIndex` (1-based current run index inside this snapshot)
   - `runTotal`
   - `runPct` (current run percent)
   - `overallPct`
   - `updatedAt`
2. Backend updates this progress state:
   - at loop run start (`runPct=0`)
   - during worker progress messages
   - on done/canceled/error terminal states
3. Backend exposes persisted progress via `GET /api/optimizer/loop/status` (`progress` field).
4. Frontend optimizer API type updated to include the new `progress` payload.
5. Frontend progress bar logic now prioritizes server `loop/status.progress.runPct` directly.
6. Added DEV-only debug instrumentation in optimizer page:
   - enabled only when `import.meta.env.DEV`
   - and `localStorage.debugOptimizerProgress === "1"`
   - throttled to max 2 logs/sec
   - logs: `jobId`, `status`, `runIndex/runTotal`, `runPct`, `overallPct`, timestamp.
7. Preserved completed 100% on restored finished job snapshot (`done` and `lastPct` restore to 100 for done jobs).
