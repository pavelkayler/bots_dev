1) Receive Data endpoints added
- Extended `backend/src/dataset/receiveDataStore.ts` to fetch real Bybit historical Open Interest via `GET /v5/market/open-interest` (`category=linear`, `limit=200`, cursor pagination) and real historical funding via `GET /v5/market/funding/history` (`category=linear`, `limit=200`, backward endTime walk loop).
- Both requests use the existing shared limiter (`500 requests / 5 seconds`) and existing fetch/retry flow style.

2) Kline -> OI interval mapping implemented
- Mapping function:
  - `1|3|5 -> 5min`
  - `15 -> 15min`
  - `30 -> 30min`
  - `60|120 -> 1h`
  - `240|360|720 -> 4h`
  - `D|W|M -> 1d`

3) Cache row alignment logic
- Receive Data now fetches OI and funding series once per symbol for the full requested range.
- Each candle row aligns with latest sample at or before candle end:
  - `candleEndMs = startMs + intervalToMs(interval) - 1`
  - OI: latest `timestamp <= candleEndMs`
  - funding: latest `fundingRateTimestamp <= candleEndMs`
- Alignment uses monotonic pointers over sorted series (`O(n)` merge behavior over candles).
- JSONL cache rows now include optional fields when available:
  - `oi`, `oiTs`, `fundingRate`, `fundingTs`

4) Dataset history metadata
- Extended `DatasetHistoryRecord` and persisted `meta.json/index.json` with:
  - `hasOi: boolean`
  - `hasFunding: boolean`
- Flags are set true when a receive job populated at least one candle row with aligned OI/funding.
- `paramsKey` remains unchanged (`universeId|startMs|endMs|interval`).

5) Optimizer cache replay updates
- In `backend/src/optimizer/runner.ts` cache mode:
  - Funding now uses real row value: `fundingRate = Number(row.fundingRate) || 0`.
  - OIV now prefers real OI: `openInterestValue = Number(row.oi) * Number(row.close)` when present.
  - Previous turnover/volume-derived OIV remains as fallback only when OI is absent.
- Funding-sign direction gating remains intact (`requireFundingSign: true`).
- Optimizer now forces `paper.applyFunding = false` for candidate runs, so funding is not charged in optimizer PnL.

6) Dev diagnostics
- Added `DEBUG_OPT_MARKETDATA=1` diagnostics in optimizer runner:
  - start/load log includes interval, sample symbol, candlesRead, oiSamplesUsed, fundingSamplesUsed, and coverage percentages.
  - run summary log includes placedOrders, filledOrders, closedTrades.

- Build checks:
  - `cd backend && npm run build`
  - `cd frontend && npm run build`
- Runtime smoke (attempted):
  - Started backend, triggered `/api/data/receive` for small 3-symbol temporary universe and short range.
  - In this environment, outbound fetch to Bybit failed (`receive_failed: fetch failed`), so live endpoint population could not be completed here.
- Once network access to Bybit is available, verify:
  - cache rows under `backend/data/cache/bybit_klines/<interval>/<symbol>.jsonl` include `oi/oiTs/fundingRate/fundingTs` for many candles;
  - optimizer run with `DEBUG_OPT_MARKETDATA=1` shows non-zero funding/OI usage where data exists;
  - optimizer PnL excludes funding fees due to forced `applyFunding=false`.
