# Stage 1 — Numeric Validation Hardening

## 1) Scope applied
Focused hardening for numeric input safety on both backend and frontend without broad refactors.

## 2) Backend changes
1. `POST /api/config` no longer silently clamps invalid numeric values for paper timing fields.
   - `paper.rearmDelayMs` must be a non-negative integer.
   - `paper.entryTimeoutSec` must be an integer `>= 1`.
   - Invalid values now return `400 invalid_config` with clear messages.
2. `POST /api/dataset-target` now validates numeric range/interval payloads and returns `400` on invalid manual ranges or unsupported intervals.
3. `startReceiveDataJob` validation hardened:
   - rejects unsupported `interval` with `invalid_interval`.
   - rejects malformed/invalid `range` with `invalid_range`.
4. Optimizer ranges backend parser tightened:
   - `priceTh`, `oivTh`, `tp`, `sl`, `offset` must be non-negative if provided.

## 3) Frontend changes
1. Optimizer start-loop validation strengthened before API call:
   - candidates: `1..2000`
   - seed: `>= 0`
   - minTrades: `>= 0`
   - optTfMin: `>= 15`
   - loopRunsCount: `>= 1`
   - marginPerTrade: `> 0`
   - leverage: `>= 1`
   - clear user-facing error text on invalid values.
2. Optimizer range validation updated:
   - `priceTh/oivTh/tp/sl/offset` min/max cannot be negative.
3. Config panel numeric parsing hardened:
   - non-negative / bounded checks applied for core strategy/execution values.
   - apply/save now reports explicit invalid numeric state instead of generic behavior.
4. Universe builder form:
   - rejects negative turnover/volatility values early with clear message.
