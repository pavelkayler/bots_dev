1) What changed per file
- `frontend/src/pages/optimizer/OptimizerPage.tsx`
  - Unlocked optimizer ranges so only minimum floors are enforced for `timeoutSec` and `rearmSec`; max values remain editable.
  - Replaced UI range key from `rearmMs` to `rearmSec`, updated defaults to seconds, and converted seconds to milliseconds at payload build time.
  - Re-enabled `tf (opt)` selector with values `15/30/60/120/240`, persisted selection, and coerced stored values below 15 up to 15.
  - Updated loop results rendering to display `rearmSec` (derived from `rearmMs / 1000`) in main results and history details tables.
  - Added `Reset` action for loop rows and wired the same reset flow to run automatically before `Start loop` request is sent (clears rows, tracked state, local draft storage, and page index).
- `frontend/src/features/optimizer/api/optimizerApi.ts`
  - Expanded optimizer range payload typings to include optional `rearmSec` for client compatibility while backend canonical field remains `rearmMs`.
- `backend/src/api/http.ts`
  - Relaxed optimizer tf validation from exact `15` to integer range `>= 15` and existing upper bound (`<= 240`) for both run and loop entrypoints.
  - Removed forced fixed range overrides and now accepts provided ranges after validation.
  - Added minimum range guards: `timeoutSec >= 61`, `rearmMs >= 900000`.
  - Kept canonical parsing on `rearmMs`; `rearmSec` remains ignored server-side.
- `backend/src/optimizer/runner.ts`
  - Removed hard-fixed optimizer constants usage for tf/timeout/rearm exact values.
  - Switched to minimum-clamp behavior only:
    - `optTfMin = max(args.optTfMin ?? 15, 15)`
    - `timeoutSec = max(candidateTimeoutSec, 61)`
    - `rearmMs = max(candidateRearmMs, optTfMin * 60_000)`
  - Preserved paper-like replay and signal model behavior (close-only events, funding sign gating, OIV based only on true OI).

2) Why
- To restore user control over optimization search space maxima while keeping required floor constraints.
- To make rearm configuration/operator review human-readable in seconds while preserving backend millisecond internals.
- To allow optimizer timeframe exploration at 15m and above without accepting too-small tf values.
- To prevent stale loop results from leaking across restarts/new loop runs.

3) How to verify locally (commands + UI)
- Build backend:
  - `npm --prefix backend run build`
- Build frontend:
  - `npm --prefix frontend run build`
- API checks:
  - Send `/api/optimizer/loop/start` with `optTfMin: 30` and valid ranges (`timeoutSec >= 61`, `rearmMs >= 900000`) and verify accepted.
  - Send `/api/optimizer/loop/start` with `optTfMin: 5` and verify `400 invalid_opt_tf_min`.
  - Send ranges with `timeoutSec.min: 60` or `rearmMs.min: 600000` and verify `400 invalid_optimizer_run_payload`.
- UI checks:
  - On Optimizer page, verify tf selector is enabled and offers `15/30/60/120/240`.
  - Verify Ranges table shows `rearmSec` and max inputs for `timeoutSec`/`rearmSec` are editable.
  - Click `Start loop` and verify results table resets immediately before new data arrives.
  - Click `Reset` in table actions and verify table clears and remains cleared after reload unless new rows arrive.

4) Known limitations
- Replay remains close-only (one ticker per candle close + close-time kline confirm).
- Funding history remains sparse and hold-last-known for alignment when points are unavailable.
- Optimizer mode still excludes funding fee transfer from realized PnL by design.
