# 72 - Risk counting + recorder universe consistency hardening

Date: 2026-03-07

## 1) Issue summary
Found and fixed critical consistency gaps:
- `maxTradesPerDay` was incremented by both `ORDER_FILLED` and `POSITION_OPEN` in paper flow.
- Recorder universe selection did not fully drive WS subscription scope and write filtering.
- Recorder process status exposed a fake `0%` progress value.
- Frontend process-status type still allowed stale recorder mode variant.

## 2) Runtime risk-limit fix
File:
- `backend/src/runtime/runtime.ts`

Change:
- Daily trades counter now increments only on actual open events:
  - `POSITION_OPEN`
  - `DEMO_POSITION_OPEN`
- `ORDER_FILLED` no longer consumes daily trade budget.

Tests:
- `backend/src/tests/runtimeRiskLimits.test.ts`
  - updated to assert that placement/fill do not consume budget.
  - budget increments only after open event.

## 3) Recorder universe + WS coordination fix
Files:
- `backend/src/api/wsHub.ts`
- `backend/src/recorder/MinuteOiRecorder.ts`
- `backend/src/tests/minuteOiRecorder.test.ts`

Changes:
- WS topic target now resolves from active mode/state:
  - `record_only`: recorder universe symbols
  - `record_while_running` + runtime active: union of trading universe and recorder universe
  - runtime without recorder: trading universe
- Stream lifecycle now reconnects when effective subscription set changes.
- Recorder ingest now enforces tracked-symbol filtering.
- Recorder status exposes `waiting` state when mode is enabled but recorder is not active.

Tests:
- added recorder test for tracked-symbol filtering.
- added recorder mode test assertion for `waiting` state.

## 4) Process-status contract/UI alignment
Files:
- `backend/src/api/http.ts`
