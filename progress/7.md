# 7) Live optimizer rows: incremental table updates without refresh

## 1. What was broken
During active optimizer execution, the main results table in `/optimizer` was not accumulating rows in real time. Rows appeared only after page refresh, even though backend persistence had already captured results.

## 2. Why this manifested only until page refresh
- The backend worker was reporting progress snapshots (`previewResults`) but not sending append-only row batches through websocket.
- Frontend relied heavily on polling and occasional result replacements, so many intermediate server-side rows were not appended live into the displayed dataset.
- On refresh, the UI rehydrated from persisted backend job snapshot/results, so rows suddenly appeared.

In short: persistence worked, but live transport + append merge path for incremental rows was missing.

## 3. What we changed

### 3.1 Backend incremental row pipeline
1. Added worker-side append batching/throttling:
   - New worker message: `rows_append` with `{ jobId, rows }`.
   - Flush policy: batch size or throttle window (200ms / 50 rows).
2. Added runner hook `onRowsAppend` and emit each accepted candidate result immediately into append flow.
3. Added main-thread merge logic in optimizer job state:
   - Merge append rows by stable params signature.
   - Deduplicate by signature and keep latest row.
   - Preserve persisted in-memory job rows during run.
4. Added WS transport:
   - New WS message: `optimizer_rows_append` with `{ jobId, rows }`.
   - Extended WS snapshot payload with optimizer snapshot `{ jobId, rows }` for reconnect/refresh hydration.
5. Wired HTTP optimizer route to register optimizer snapshot provider with wsHub.

### 3.2 Frontend live append behavior
1. Extended websocket-lite hook to forward `lastMsg` updates in lite listeners.
2. Added optimizer WS message handling in `OptimizerPage`:
   - On `optimizer_rows_append`, append rows cumulatively for active job using functional `setState` and dedupe map keyed by params signature.
   - Throttle resort/render work to 200ms.
3. Added snapshot hydration handling:
   - On WS `snapshot`, if it contains active optimizer job rows, hydrate current rows from persisted snapshot.
4. Loop mode compatibility:
   - Loop run rows are merged into per-run aggregate (`__runJobId`) without fabricating rows.
5. DEV instrumentation added:
   - Guarded by `import.meta.env.DEV` and `localStorage.debugOptimizerRows === "1"`.
   - Logs at most 2x/sec with `{ jobId, batchSize, totalDisplayed }`.

## 5. How to verify
1. Start backend and frontend.
2. Open `/optimizer`, choose dataset history rows, start optimizer loop or single run with a small candidates count (e.g., 100).
3. Confirm table starts filling before run completes (no refresh).
4. Mid-run, refresh page:
   - Table should rehydrate from snapshot rows.
   - New rows should continue appending via `optimizer_rows_append`.
5. Optional DEV logging:
   - In browser console: `localStorage.debugOptimizerRows = "1"` and rerun.
   - Observe throttled `[optimizer-rows]` logs (~2x/sec max).

## 6. What we tried before and why it failed
Earlier loop-start `400` failures were caused by stale `datasetTarget` references inside `/api/optimizer/run` when switching to dataset history ids. That issue has already been fixed and was **not** touched here.

This fix avoids reintroducing that bug by keeping dataset history selection flow intact and only adding live row append transport/merge behavior.
