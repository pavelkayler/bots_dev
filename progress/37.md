2) Split definition and rationale
- Split is computed from `[rangeStartMs, rangeEndMs]` as:
  - `total = rangeEndMs - rangeStartMs`
  - `splitMs = rangeStartMs + floor(total * 0.70)`
  - then rounded down to minute boundary.
- Train window: `[rangeStartMs, splitMs]`
- Validation window: `[splitMs, rangeEndMs]`
- This surfaces parameter stability across later unseen time while keeping deterministic, low-complexity evaluation.

3) How to verify locally
- Build:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- Run optimizer on one dataset history row with small candidate count (for example, 20).
- Confirm each emitted row includes finite values for:
  - `trainNetPnl`, `trainTrades`, `valNetPnl`, `valTrades`, `valPnlPerTrade`
- In Optimizer table, click "val pnl/trade" and verify sort order changes correctly.
- Re-run with different selected time range length and confirm split metrics shift due to changed split boundary.

4) Known limitations
- Compute cost is approximately 2x versus single-run per candidate because each candidate runs train + validation replays.
- This is still a single holdout split, not full walk-forward or k-fold time-series validation.
