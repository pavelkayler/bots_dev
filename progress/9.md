# Progress Report 9

## 1) Root cause analysis: why no positive trades after tape removal

1. In the Receive Data pipeline, Bybit REST kline requests were hardcoded to `interval=1`, and cache rows were written without storing interval context. This made the system effectively single-timeframe even though optimizer settings could vary.
2. In the optimizer dataset loader (`runOptimizationCore` cache branch), synthetic tape metadata was also hardcoded to `klineTfMin: 1` and event timestamps were hardcoded as `startMs + 60_000` for every candle.
3. This created timeframe mismatch behavior: if non-1m data is selected/expected, strategy candle bucketing (`tfMs`) and fallback ref windows become inconsistent with actual candle spacing, which can suppress valid entries and skew outcomes.
4. Additional risk was cache collisions: all cached klines were in one file per symbol (`.../bybit_klines/<SYMBOL>.jsonl`), so future mixed TF runs would blend datasets.

### Evidence hooks added
- Backend DEV log (`DEBUG_DATASET_TF=1`) during optimizer dataset stream build logs one sampled symbol with:
  - `interval`
  - `tfMin`
  - `firstTs`
  - `lastTs`
  - `candleCount`
- Frontend DEV logs (guarded by `localStorage.debugDatasetTf === "1"`) show selected interval and intervals returned in history rows.

## 2) Changes implemented

### Backend
1. Added dataset target `interval` support and normalization with allowed Bybit V5 values:
   - `1, 3, 5, 15, 30, 60, 120, 240, 360, 720, D, W, M`
   - Default remains `"1"`.
2. Receive Data now sends selected interval to `/v5/market/kline` and computes batch span by interval size.
3. Cache is now separated by timeframe:
   - `backend/data/cache/bybit_klines/<INTERVAL>/<SYMBOL>.jsonl`
4. Graceful migration:
   - for interval `"1"`, reads fallback to legacy path `backend/data/cache/bybit_klines/<SYMBOL>.jsonl` if interval folder file is missing.
5. Dataset history metadata now stores `interval` in `meta.json` and index record.
6. `paramsKey` updated to include interval:
   - `universeId|startMs|endMs|interval`
7. Optimizer run + loop endpoints now:
   - require at least one history id
   - verify selected histories exist
   - reject mixed intervals with 400 and clear message
   - validate cache files under interval-specific cache path (with interval=1 legacy fallback)
   - pass interval into optimizer run payload.
8. Optimizer dataset loader now uses interval-aware cache path and interval-aware synthetic timestamps (`startMs + intervalMs`) + `klineTfMin` derived from interval.
9. Kept chronological processing behavior:
   - selected history rows sorted by `(startMs, endMs, receivedAtMs)`
   - per-symbol windows sorted/merged
   - cache rows loaded in ascending timestamp order.

### Frontend
1. Added **Timeframe** selector to Dataset Target UI with allowed Bybit values.
2. Persisted timeframe in localStorage as part of existing dataset target draft state.
3. Receive Data payload now includes timeframe.
4. Dataset history table now includes sortable `tf` column.
5. Added DEV-only console logs for timeframe debugging.

## 3) Migration behavior

1. Existing installs with old cache layout are preserved for 1m:
   - reader checks new path first (`/1/<symbol>.jsonl`), then old path (`/<symbol>.jsonl`).
2. Existing history rows without interval are normalized as `"1"` during reads.
3. New history rows always store explicit interval.

## 4) Verification steps

1. Build backend:
   - `npm --prefix backend run build`
2. Build frontend:
   - `npm --prefix frontend run build`
3. Manual check in UI:
   - open Optimizer page
   - in Dataset Target, select timeframe and run Receive Data
   - verify new history row shows selected `tf`
4. Mixed-timeframe validation:
   - select history rows with different `tf`
   - start optimizer run/loop
   - confirm 400 with message: selected rows must have same timeframe.
5. Dev evidence logs:
   - backend with `DEBUG_DATASET_TF=1`
   - frontend with `localStorage.debugDatasetTf = "1"`

## 5) What we tried before and why it failed

1. Earlier iterations around history/export UX (export button flow and history table reshaping) improved operator workflow but did not address dataset temporal semantics.
2. Those iterations did not propagate timeframe end-to-end (UI → Receive Data → cache layout → history metadata → optimizer loader), so the optimizer still consumed data under 1m assumptions.
3. This change is different because it fixes the data contract itself and adds proof logs at the exact dataset stream construction boundary used by optimizer execution.
