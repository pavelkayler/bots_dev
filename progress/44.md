1) What changed
- Updated `backend/src/optimizer/runner.ts` to support multi-run candidate generation inside one optimization call via optional `runsCount`.
- Added a loop-session set: `seenCandidateKeys`, scoped across all runs in that call.
- Switched candidate generation to run-specific deterministic RNG streams: `effectiveSeed = effectiveSeedBase + runOffset`.
- Candidate evaluation now uses per-run `evaluated/attempts` with `maxAttempts = candidates * 100`.
- Candidate keys are computed immediately after draw+clamp and checked before simulation:
  - skip if already present in `seenCandidateKeys`
  - skip if `rememberNegatives` blacklist contains the key
- Added duplicate-skip diagnostics to checkpoint payload: `duplicatesSkippedTotal`.
- Kept rows append streaming and final result shape unchanged.

2) Why duplicates happened and how this fix works
- Previously, optimization evaluated from a single RNG stream for one run scope and had no session-wide dedupe set. In looped operation with repeated seeds, identical parameter sets could be simulated repeatedly.
- This fix prevents re-simulation of the same `candidateKey` within the same multi-run session by tracking all seen keys in `seenCandidateKeys`.
- Using deterministic per-run seeds (`base + runOffset`) expands search coverage across runs while remaining deterministic.
- `rememberNegatives` behavior is preserved: blacklisted negatives are still skipped exactly as before.

3) How to verify
- Build checks:
  - `npm --prefix backend run build`
  - `npm --prefix frontend run build`
- Runtime check with fixed seed (example: `runsCount=3`, `candidates=200`):
  - Start optimizer loop and inspect combined output candidate keys/params.
  - Confirm no repeated `candidateKey` is simulated within that loop session.
  - Confirm checkpoint summaries include `duplicatesSkippedTotal` (typically > 0 in constrained search spaces).
  - Confirm each run evaluates up to `candidates` unless attempts cap is hit.

4) Known limitations
- If candidate space is constrained and many draws collide, `attempts` can hit `maxAttempts` (`candidates * 100`) before filling all requested candidates; in that case evaluated candidates can be `< candidates` for that run.
