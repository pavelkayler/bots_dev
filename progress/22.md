1) What changed per file
- `backend/src/dataset/receiveDataStore.ts`
  - Receive Data now always ensures only 1m kline cache coverage for requested range (`bybit_klines/1`).
  - Added incremental 5m open-interest cache handling (`bybit_open_interest/5min`) and incremental funding history cache handling (`bybit_funding_history`).
  - Added funding history REST fetch (`/v5/market/funding/history`) and persisted cache rows as `{timestamp,fundingRate}`.
  - Updated progress accounting so `totalSteps/completedSteps` reflects fetch windows actually executed (kline + OI + funding).
  - Aligned OI into 1m kline rows (`row.oi`, `row.oiTs`) using latest OI sample with timestamp `<=` candle close.
  - Kept interval compatibility: if request omits interval, backend defaults to `"15"` while receive behavior remains 1m+5m+funding.
- `backend/src/api/http.ts`
  - `/api/optimizer/run` now enforces `optTfMin === 15` (rejects any other value, including tampered payloads).
  - `/api/optimizer/loop/start` now enforces `optTfMin === 15` as well.
  - Forced optimizer run payload ranges for `timeoutSec` and `rearmMs` to fixed values (`61`, `900000`) server-side.
- `backend/src/optimizer/runner.ts`
  - Added fixed optimizer constants: `optTfMin=15`, `timeoutSec=61`, `rearmMs=900000` and applied them defense-in-depth in runner.
  - Kept close-only replay (single ticker at candle close + close-time `kline_confirm`).
  - Restored funding-direction gating from cached funding history:
    - loads per-symbol funding history cache,
    - uses latest sample with `timestamp <= tick ts`,
    - passes fundingRate into `SignalEngine.decide()` and `PaperBroker.tick()`,
    - enables `requireFundingSign=true` for optimizer SignalEngine.
  - Funding fee remains excluded from PnL (`applyFunding: false` unchanged).
  - Ensured OIV is derived only from real OI (`oiBase * close`) with no volume/turnover fallback.
- `frontend/src/features/datasetTarget/ui/DatasetTargetCard.tsx`
  - Removed timeframe selector from Receive Data UI.
  - Removed draft interval state from card; card now keeps Universe + Range only.
  - For compatibility payload, still sends constant `interval: "15"` silently.
- `frontend/src/pages/optimizer/OptimizerPage.tsx`
  - Optimizer TF is fixed to `15` and rendered as disabled single-value control.
  - Removed optimizer TF localStorage read/write behavior.
  - Locked range defaults for timeout/rearm to fixed values (`61`, `900000`) and disabled their range inputs in UI.
  - Loop start payload now always sends `optTfMin: 15`.

2) Why
- Changes intentionally lock behavior to paper-like constraints (not “better-than-paper”):
  - fixed 15m optimizer TF,
  - close-only replay,
  - fixed timeout/rearm floors and effective values,
  - funding used only as direction gate (no funding PnL transfer).
- Receive Data now matches required data dependencies for replay/parsing:
  - full 1m candle coverage,
  - full 5m OI coverage,
  - funding history coverage (sparse points acceptable).
- Incremental fetch avoids redundant downloads and allows fast completion when cache is already complete.

3) How to verify locally
- Backend build:
  - `npm --prefix backend run build`
- Frontend build:
  - `npm --prefix frontend run build`
- API checks:
  - POST `/api/optimizer/run` with `optTfMin: 30` should return `400 invalid_opt_tf_min`.
  - POST `/api/optimizer/loop/start` with `optTfMin: 5` should return `400 invalid_opt_tf_min`.
  - Run Receive Data twice for same Universe+Range and confirm second run quickly completes with near-zero fetch windows.
- UI checks:
  - Dataset Target card shows Universe + Range controls only (no timeframe picker).
  - Optimizer page shows TF fixed at 15 (disabled).
  - Timeout/Rearm rows visible in ranges table but disabled and fixed at 61 / 900000.
  - Results table still includes direction mode column.

4) Known limitations
- Funding history endpoint is sparse by funding schedule; replay uses hold-last-known sample and does not synthesize extra points.
- Cooldown around funding windows is not introduced in this change.
- Funding fee is intentionally not applied to PnL in optimizer mode.
